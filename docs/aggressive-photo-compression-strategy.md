# Aggressive Photo Compression Strategy

Date: 2026-06-28
Branch: `audit/aggressive-photo-compression-strategy`

This report started as an audit. The approved follow-up implementation keeps export behavior simple and changes only future image storage behavior plus internal diagnostics.

## Executive Summary

Inspectley already has several important storage/export safeguards:

- Imported photos are saved under `Documents/InspectorPro/Images/` as metadata-stripped JPEGs, not as unlimited raw camera originals.
- Imported images are resized to `AppConstants.importMaxDimension` (`2000`) before save.
- Export uses `FlattenedExportImageRenderer` and `ImageCompressor` to generate compressed JPEG payloads before PDF/DOCX output.
- DOCX embeds those compressed JPEG files directly into `word/media`.
- Exports are transient in practice: the app deletes the shared export file after the share sheet completes and purges leftover exports on launch.

The main remaining risks are:

- Import resize was width-based; it has now been changed to long-edge/max-dimension based for future imports.
- Annotated photos are saved as a second JPEG. New annotated images now use quality `0.85` instead of the previous `0.92`.
- Thumbnails are memory-only and generated by loading full display images first; no persisted thumbnail file exists in the active schema.
- Current export UI intentionally keeps `.economical` as the default and does not expose quality choices.
- The product decision is to avoid export file-size display, warning popups, and extra export decisions for now.
- PDF decodes compressed export JPEGs back into `UIImage` and draws them into the PDF; DOCX embeds the compressed JPEG bytes directly. PDF final size must be measured because UIKit controls the final PDF image encoding.

Updated product decision:

- Keep export economical/lightweight by default.
- Do not add Standard / High Quality / Email Friendly modes.
- Do not add an export quality picker.
- Do not add export size warnings or file-size display UI.
- Focus this phase on reducing future app storage growth.

## Current Implementation Findings

### Image Storage

Original photo location:

- Base directory is `Documents/InspectorPro/Images/` from `AppConstants.imagesBaseURL`.
- The app base, images, branding, and exports directories are ensured at launch in `FileManagerService.ensureDirectoriesExist()`.
- File references stored in SwiftData are relative paths, e.g. `<project UUID>/<photo UUID>.jpg`.

Relevant files:

- `InspectorPro/Utilities/Constants.swift:4-35`
- `InspectorPro/Services/FileManagerService.swift:9-20`
- `InspectorPro/Services/ImageStorageService.swift:6-29`

Original photo format and compression:

- `ImageStorageService.saveImage` resizes the input `UIImage` with `image.resized(maxDimension: AppConstants.importMaxDimension)`.
- It writes `jpegDataStripped(quality: AppConstants.importJPEGQuality)`, currently `0.85`, as `<uuid>.jpg`.
- Metadata is stripped by recreating the image from `cgImage` with orientation `.up`.
- This is JPEG, not PNG.

Relevant files:

- `InspectorPro/Services/ImageStorageService.swift:19-27`
- `InspectorPro/Utilities/Constants.swift:9-15`
- `InspectorPro/Utilities/UIImage+Extensions.swift:4-25`

Historical nuance:

- The old import path used `resized(maxWidth:)`, which allowed portrait/tall images to remain above the intended 2000px long edge.
- The current import path uses max dimension / long edge and preserves aspect ratio.

Annotated photo storage:

- `PhotoRecord.displayImagePath` prefers `annotatedImagePath` when that file exists, otherwise falls back to `imagePath`.
- `ImageStorageService.saveAnnotatedImage` writes a flattened annotated composite to `ann_<originalUUID>.jpg`.
- Annotated images use `jpegDataStripped(quality: AppConstants.annotatedJPEGQuality)`, currently `0.85`.
- The original image remains on disk. Annotated reports can therefore store both the original JPEG and the annotated JPEG for one logical photo.

Thumbnails:

- `ThumbnailService` keeps an in-memory `[String: UIImage]` cache with a limit of 300 entries.
- It loads the full display image through `ImageStorageService.loadImage`, then creates a 200px `UIImage` thumbnail in memory.
- `AppConstants.thumbnailJPEGQuality` exists but is not used by the active thumbnail path.

### Export Behavior

Export entry point:

- `ExportEngine.exportReport` dispatches to `PdfExporter` or `DocxExporter`.
- Empty reports are blocked with `ExportError.noPhotos`.

Relevant file:

- `InspectorPro/Export/ExportEngine.swift:25-71`

Current export UI:

- The user can choose only format: `PDF` or `Word (DOCX)`.
- `selectedFormat` currently defaults to `.docx`.
- Quality is hardcoded to `.economical` in `ExportOptionsSheet.startExport`.
- There is no quality/mode picker, no file size estimate, and no large-attachment warning.

Current export image pipeline:

- `FlattenedExportImageRenderer.render(photo:options:)` uses `photo.displayImagePath`, so annotated photos export the annotated JPEG.
- It reads the chosen source file from `Images/`.
- It calls `ImageCompressor.compressData` with `options.exportImageMaxRenderWidth` and `options.exportImageMaxBytes`.
- `ImageCompressor` decodes the source, resizes by max width, then tries to meet the byte budget by lowering quality, reducing width, then lowering quality further.
- It uses `autoreleasepool` per compression operation.

Relevant files:

- `InspectorPro/Export/FlattenedExportImageRenderer.swift:20-63`
- `InspectorPro/Export/ImageCompressor.swift:3-114`
- `InspectorPro/Export/ExportOptions.swift:119-137`

Current default economical export parameters for large reports:

| Photo count | Approx render width from current code | Per-image byte cap |
|---:|---:|---:|
| 50 | 629px | 100KB |
| 100 | 596px | 80KB |
| 300 | 530px | 55KB |
| 500 | 500px | 55KB |

These values come from the current A4 table image column width and `.economical` adaptive logic. They are very storage-friendly but below the requested 900-1200px professional default range.

PDF export:

- PDF prepares all flattened/compressed export images before rendering the PDF.
- It decodes compressed JPEG bytes into `UIImage` and draws each image into the table cell.
- It does not draw the original full-size storage image.
- It draws images directly into the cell rectangle, so the current table image draw path does not preserve aspect ratio.
- Because `UIGraphicsPDFRenderer` receives a `UIImage`, final PDF image encoding is controlled by UIKit/Core Graphics and should be measured with real exports.

Relevant files:

- `InspectorPro/Export/PdfExporter.swift:15-24`
- `InspectorPro/Export/PdfExporter.swift:33-89`
- `InspectorPro/Export/PdfExporter.swift:355-397`

DOCX export:

- DOCX creates a temp package directory and `word/media`.
- For each photo, `processImage` calls `FlattenedExportImageRenderer.render`, writes `exportImage.data` as `image<relId>.jpg`, and references it in document relationships.
- DOCX therefore embeds the compressed JPEG payload directly.
- DOCX image display extent is set to the table image content area, and `ImageCrop.none` is used.

Relevant files:

- `InspectorPro/Export/DocxExporter.swift:28-47`
- `InspectorPro/Export/DocxExporter.swift:92-131`
- `InspectorPro/Export/DocxExporter.swift:220-244`

Memory behavior:

- DOCX processes one image per loop and writes it to temporary media.
- PDF currently builds an array of every `FlattenedExportImage` before creating the PDF. For 300-500 photos this can still be meaningful memory pressure even after compression.
- Import saves every `AppConstants.importSaveCheckpoint` photos, currently every 20 photos.

## Current Risks

### High

- **Annotated storage can double photo disk usage.** Original and annotated JPEGs both exist. Future annotated images now use quality `0.85`, but heavily annotated reports will still store two JPEGs for those photos.
- **Large reports still multiply storage quickly.** A 300-500 photo report can still consume substantial storage even with bounded imports, so real-device sample checks remain important.

### Medium

- **No persistent thumbnails.** Lists avoid displaying full-size images by using `ThumbnailView`, but the service still loads the full display JPEG to generate the memory thumbnail.
- **PDF memory profile is weaker than DOCX.** PDF precomputes every compressed export image before writing.
- **Compression byte caps are best effort.** `ImageCompressor` can still return data above `maxBytes` after reaching minimum width or quality.
- **PDF final image encoding is not explicit.** The pipeline feeds a compressed-size `UIImage` into the PDF renderer, but the final PDF stream encoding should be measured.

### Low

- **Unused/legacy thumbnail model fields exist in older schemas.** Active schema does not use `thumbnailPath`, but old migration code still carries legacy fields.
- **Quality preset labels are misleading for current UI.** Localized strings mention 900/1400/2000px descriptions, but the export sheet does not expose those presets and current economical render width can be much lower.

## Recommended Compression Settings

### A. In-App Working Image

Goal: preserve enough detail for viewing and annotation without keeping huge camera originals.

Recommended future policy:

- Store JPEG only.
- Resize by long edge / max dimension, not width only.
- Long edge: 1600-2000px.
- JPEG quality: 0.75-0.85.
- Strip metadata.
- Preserve one original working image per logical photo.
- For annotated images, either keep one flattened annotated JPEG per photo or consider storing vector annotations in a future schema so the annotated JPEG can be regenerated.

Recommended initial change later:

- Replace import `resized(maxWidth:)` usage with a max-dimension helper.
- Keep default at 2000px / quality 0.80-0.85 until real sample QA proves a lower value is acceptable.

### B. Export Image

Goal: keep export simple and lightweight.

Current product policy:

- Keep `.economical` as the default export quality.
- Do not add an export mode picker.
- Do not add file-size display or warning popups.
- Do not add Standard / High Quality / Email Friendly modes.
- Preserve the current PDF/DOCX export UX unless a separate export-specific task approves changes.

Future export tuning, if any, should happen invisibly inside the economical pipeline and should be verified with real 50/100/300/500 photo reports.

### C. Thumbnail Image

Goal: make list/grid UI cheap and avoid repeated full-image loads.

Recommended future policy:

- Generate persisted thumbnails when importing or when saving an annotated image.
- Thumbnail long edge: 300-500px.
- JPEG quality: 0.55-0.70.
- Store under a separate thumbnail folder or sibling naming scheme.
- Add `thumbnailPath` to the next SwiftData schema version if persisted thumbnail tracking is needed.
- Delete thumbnail files with original/annotated photo cleanup.

The current 200px memory thumbnail is efficient after it exists in RAM, but too small for future richer grids and does not avoid initial full-image decode.

## Expected File Sizes

These are planning estimates, not measured app outputs. Actual sizes depend on image complexity, portrait/landscape mix, annotations, PDF encoder behavior, Word package compression, and logo/footer content.

Assumptions:

- Current app storage: imported originals around 0.45-1.35MB/photo; annotated reports can roughly double that. The table uses 0.45-2.7MB per logical photo to include mixed original/annotated reports.
- Current export: hardcoded economical mode, around 55-100KB/photo before container overhead, with PDF/DOCX overhead added.

| Photos | Current/future app storage before this storage optimization | Current lightweight PDF export | Current lightweight DOCX export |
|---:|---:|---:|---:|
| 50 | 23-135MB | 5-11MB | 5-13MB |
| 100 | 45-270MB | 7-14MB | 7-16MB |
| 300 | 135-810MB | 16-28MB | 16-30MB |
| 500 | 225MB-1.35GB | 26-44MB | 26-46MB |

Interpretation:

- Current export may already be small enough for some 50-100 photo email workflows because it intentionally stays economical/lightweight.
- 300-500 photo reports are inherently difficult for email. Even aggressively compressed outputs can exceed common attachment limits.
- In-app storage can still become very large for fully annotated reports because every annotated photo keeps both original and annotated JPEGs, but future imports and annotated images are now bounded more tightly.

## Email Feasibility

Email attachment reality:

- Under 20MB: usually email safe.
- 20-25MB: borderline.
- Over 25MB: not reliable for email.

Expected feasibility:

- 50 photos: usually email-safe with current economical export unless image content is unusually complex.
- 100 photos: plausible but can become borderline.
- 300 photos: generally not email-safe unless extremely aggressive compression is used, and quality must be checked carefully.
- 500 photos: not realistically reliable for email at professional quality.

Product guidance:

- Keep the export flow simple and avoid adding warning popups or file-size UI.
- If a user cannot send a very large report through email, that can be addressed through support/documentation or a separately approved sharing workflow later.
- Do not add extra export decisions in this phase.

## Quality Check Recommendation

Before changing defaults, run real sample reports:

- Generate 50, 100, 300, and 500 photo reports.
- Include wall cracks, leaks, mold, electrical defects, serial labels, handwriting-like markings, and red/yellow annotations.
- Include Hebrew text in project/report metadata and defect notes.
- Export the current default lightweight PDF and DOCX.
- Compare file sizes and visual quality on iPhone and desktop.
- Check whether annotations, small cracks, mold patches, moisture stains, electrical labels, and Hebrew text remain readable.
- Confirm PDF and DOCX open without repair prompts and preserve RTL layout.
- Choose defaults based on measured output, not theoretical byte targets alone.

## Final Recommendation

Inspectley should keep export decisions simple for users. The current default export remains economical/lightweight, with no quality picker and no file-size warning UI.

For in-app storage, continue avoiding raw camera originals, use long-edge import resizing, keep import JPEG quality at `0.85`, and save future annotated JPEGs at `0.85`. Do not recompress existing user images or migrate storage unless a separate implementation plan is approved and sample quality checks pass.

This follow-up implementation changes only future image storage behavior and internal diagnostics. It does not modify existing user images.
