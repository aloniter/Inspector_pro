# Export Best-Practice Implementation Summary

## Branch

Implemented on `export-best-practice-implementation`.

## Files Changed

- `InspectorPro/Export/ExportImageGeometry.swift`
  - New shared geometry helper for aspect-fit sizing, width-fill-without-crop placement, centered placement, normalized point mapping, and point/EMU conversion.
- `InspectorPro/Export/FlattenedExportImageRenderer.swift`
  - New final export image pipeline that selects the canonical image source, compresses deterministically, and returns the final JPEG bytes plus decoded image metadata.
  - Keeps the export image as the existing original or annotated image. It does not add backgrounds, canvas framing, overlays, or generated filler.
- `InspectorPro/Export/PdfExporter.swift`
  - Uses `FlattenedExportImageRenderer` for each photo.
  - Stretches the existing export image to exactly fill the table image cell, matching manual Word resize behavior.
  - Caps each PDF finding row to the two-rows-per-page budget so normal reports keep two photos per page instead of expanding to one photo per page.
- `InspectorPro/Export/DocxExporter.swift`
  - Uses `FlattenedExportImageRenderer` for each photo.
  - Writes the renderer JPEG bytes into the DOCX package.
  - Sets DOCX image extents to the full table image cell.
  - Keeps DOCX crop metadata disabled for report images.
- `InspectorPro/Export/ExportOptions.swift`
  - Restored the default report column split to 60/40 image/text.
  - Sets the normal photo row height from the available page height divided by two, preserving the two-findings-per-page layout whenever content allows.
  - Adds a DOCX-specific 18pt table layout safety budget so Word's table pagination does not add an avoidable extra page from padding/border rounding.
  - Removed the old stretch-oriented fitting helper now replaced by `ExportImageGeometry`.
- `InspectorProTests/ExportTests.swift`
  - Adds coverage for flattened image source selection and deterministic output.
  - Adds aspect-fit geometry and normalized coordinate mapping tests.
  - Verifies DOCX embeds the same renderer JPEG bytes.
  - Verifies portrait, landscape, and square DOCX images stretch to full-cell extents without crop metadata.
  - Verifies DOCX report tables do not emit `w:bidiVisual`, avoiding Word-side visual column reversal.
- `InspectorPro.xcodeproj/project.pbxproj`
  - Regenerated by `xcodegen generate` so the new export source files are part of the target.

Pre-existing dirty files were left in place and not reverted: `project.yml`, `tasks/appstore-checklist.md`, `tasks/todo.md`, and `tasks/appstore-submission/`.

## Decisions Made

- Architecture B from `EXPORT_BEST_PRACTICE_PLAN.md` was implemented: use the flattened annotated image as the canonical export source when it exists, otherwise use the original image.
- PDF and DOCX now route through the same final export renderer. DOCX stores the renderer JPEG bytes directly; PDF draws the decoded image produced from those bytes.
- Manual QA feedback changed the placement policy: report images are now stretched to the full image cell. This can distort photo proportions, but it matches manual Word resize behavior and keeps the baked annotations visible because the whole annotated JPEG is stretched as one layer.
- After manual QA feedback, DOCX table visual order is forced to match PDF by avoiding `w:bidiVisual`: image left, description right.
- The default report split remains 60 percent image and 40 percent text because findings often need readable notes.
- DOCX annotated findings continue to avoid `a:srcRect` crop metadata.

## Before vs After

Before:

- PDF compressed and decoded images independently.
- DOCX compressed images independently.
- Export image placement used full-cell stretching, which preserved the baked annotation attachment but could visually distort the photo compared with the editor.
- Fitting helpers were mixed into `ExportOptions.swift`.

After:

- PDF and DOCX use one shared export image pipeline.
- Annotated image files are preferred automatically through `PhotoRecord.displayImagePath`.
- Image placement now uses a full-cell stretch policy in both PDF and DOCX.
- PDF row height now uses the same two-findings-per-page budget as the report layout, so a page should contain two photo rows whenever the report has another finding available and the page is not the odd final page.
- DOCX image dimensions are deterministic and tested against the full-cell table geometry.
- Export geometry and final-image rendering are isolated in dedicated export files.

## Known Limitations

- The current annotation model still persists a flattened annotated JPEG, not editable annotation vectors. This implementation improves export fidelity for the current storage model but does not introduce vector re-rendering.
- PDF long-note handling now prioritizes the requested two-photos-per-page layout. A single very long note can still clip instead of splitting across multiple PDF pages; proper continuation rows remain future work.
- DOCX pagination remains partially controlled by Word's table layout engine; this implementation improves image extents and fixes the table column order, but Word can still reflow rows differently across versions.
- Full-cell stretch intentionally changes photo proportions for portrait and square images. This is now an accepted product decision to make every image fill its table rectangle.
- Manual visual QA in Apple Preview and Microsoft Word is still required before release.

## Manual QA Checklist

- Export PDF and DOCX with a portrait image and compare annotation placement against the editor.
- Export PDF and DOCX with a landscape image and compare annotation placement against the editor.
- Export PDF and DOCX with a square image and compare annotation placement against the editor.
- Verify annotations near top-left and bottom-right landmarks do not drift.
- Verify arrows, circles, freehand strokes, and text-like marks remain attached to clear photo landmarks.
- Verify long Hebrew notes do not collide with headers or footers.
- Verify reports with 20, 50, 100, and 300+ findings remain stable and reasonably sized.
- Verify stale or missing image paths still fail with the existing export error behavior.
- Verify exports with logo/footer branding and without branding.
- Verify DOCX in Microsoft Word and PDF in Apple Preview.

## Scope Exclusions

The implementation did not intentionally touch App Store, Fastlane, screenshots, login/auth, Supabase logic, branding settings UI, or unrelated app areas.
