# Project Overview

**Inspectley** is an iOS 18.0+ SwiftUI app for building inspectors to create professional inspection reports with photos. It exports to PDF and DOCX with full Hebrew/RTL support.

- Product / App Store name: **Inspectley**. Internal Xcode project, target, scheme, Swift module, and folders remain **InspectorPro**; bundle id is `com.aloniter.inspectorpro`. Do NOT rename the internal identifiers — use "Inspectley" only for user-facing/product/App Store references.
- **Production app.** Current repo version is 1.0.2 build 3 in project.yml. Actual App Store Connect live/submitted version must be confirmed manually before release.
- Platform: iOS 18.0+, iPhone only (portrait), SwiftUI + SwiftData
- Build system: Xcode project generated by XcodeGen (`project.yml` is source of truth)
- Dependencies (SPM): ZIPFoundation 0.9.19+ and supabase-swift 2.0+ (auth + export permission gating)

**Read first:** `docs/AI_CONTEXT.md` (shared deep context: flows, storage/export architecture, do-not-touch areas, verification rules) and `tasks/lessons.md` (mandatory before any export/RTL/numbering work — it documents every approach that already failed in real Word). `AGENTS.md` is the equivalent entry point for Codex/general agents and must stay aligned with this file.

# Build & Test Commands

```bash
# Build
xcodebuild -project InspectorPro.xcodeproj -scheme InspectorPro -configuration Debug build

# Run tests (uses Swift Testing, not XCTest)
xcodebuild -project InspectorPro.xcodeproj -scheme InspectorPro test

# Regenerate Xcode project from project.yml
xcodegen generate
```

Tests live in `InspectorProTests/ExportTests.swift` — 86 tests (as of 2026-07) using `#expect()` syntax (Swift Testing).

**Test-run rules** (hard-won, see `tasks/lessons.md`):
- Use a fresh `-derivedDataPath` (scratch directory) for simulator test runs — the default DerivedData can silently install a stale app binary while reporting BUILD/TEST SUCCEEDED.
- Never run two xcodebuild invocations concurrently on this project — racing DerivedData can leave an unsigned app product that breaks the simulator.

# Architecture

**3-tier structure:**

1. **Data**: SwiftData (`Project` → `Report` → `PhotoRecord`, plus `BrandingProfile` linked per report) + FileManager for disk images
   - Versioned schemas V1–V9 in `Models/InspectorProMigration.swift`; **V9 is active**. Only append V10+, never edit existing versions.
   - Images stored as JPEGs at `Documents/InspectorPro/Images/<projectID>/`; only relative paths in SwiftData
   - `PhotoRecord.displayImagePath` returns annotated path if available, else original

2. **Services**: `FileManagerService`, `ImageStorageService`, `ThumbnailService` (in-memory 200px thumbnails for grid), `AuthService`/`SupabaseManager` (login), `ExportPermissionService` (server-side export gate, 6h offline cache), `CompanyBrandingService`

3. **Views**: `ProjectListView` → `ProjectDetailView` (reports, photo import/reorder/delete) → photos; export via modal sheets. Exports are **transient**: deleted after the share sheet completes and purged at every launch.

**Key files:**
- Entry point: `InspectorProApp.swift`
- Constants & limits: `Utilities/Constants.swift` (`AppConstants`, `AppLanguage`, `AppStrings`)
- Export dispatcher: `Export/ExportEngine.swift`
- Layout constants: `Export/ExportOptions.swift` (margins, column ratios, DPI)

# Export System

```
ExportEngine → PdfExporter  (UIGraphicsPDFRenderer drawing)
             → DocxExporter (programmatic OpenXML → ZIPFoundation zip)
Shared by both formats:
  ExportOptions (ALL A4 geometry), ImageQuality (adaptive byte/width budgets),
  FlattenedExportImageRenderer + ImageCompressor (identical image bytes in
  PDF and DOCX), OpenXMLBuilder (full RTL: bidiVisual, bidi, rtl, cs font)
```

DOCX structure is **generated programmatically** in `DocxTemplateBuilder.swift` — no binary template is used for the document at all. `Resources/template.docx` has exactly one role: `TemplateExtractor` pulls the bundled default logo image out of it.

**Layout:** A4, 2 photos/page, 60% image / 40% text columns, header ~92pt, footer ~72pt.

**RTL in OpenXML:** `<w:bidiVisual/>` on tables, `<w:bidi/>` on paragraphs, `<w:rtl/>` on runs, `<w:rFonts w:cs="Arial"/>` for Hebrew.

# Important Constraints

- **Memory**: image compression runs inside `autoreleasepool` (`ImageCompressor`); photo import saves to SwiftData every 20 photos. Known open issue: `PdfExporter` accumulates all export images before rendering (see `docs/AI_CONTEXT.md`, "Known open issues").
- **Report design is frozen**: do not casually change PDF/DOCX layout, Hebrew/RTL behavior, attendee numbering, or storage code without explicit owner approval.
- **iOS 18.0 minimum**: Do not use features requiring guards for < iOS 18
- **Image paths are relative**: Always relative to `Documents/InspectorPro/Images/`
- **Localization**: Hebrew is default (RTL). Add strings to both `he.lproj/Localizable.strings` and `en.lproj/Localizable.strings`; use `AppStrings.text()` to retrieve

# Workflow Orchestration

## 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately — don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

## 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

## 3. Self-Improvement Loop
- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

## 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

## 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes — don't over-engineer
- Challenge your own work before presenting it

## 6. Autonomous Bug Fixing
- When given a bug report: just fix it. Don't ask for hand-holding
- Point at logs, errors, failing tests — then resolve them
- Zero context switching required from the user
- Go fix failing CI tests without being told how

# Core Principles

- **Simplicity First**: Make every change as simple as possible. Impact minimal code.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.
